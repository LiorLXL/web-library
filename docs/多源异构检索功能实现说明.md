# 多源异构检索功能实现说明

本文面向 GitHub 公开仓库和代码评审，介绍本项目中“多源异构检索”的功能目标、实现方式、关键接口和安全边界。文档只描述可公开的功能实现，不包含 API Key、token、本地演示路径或个人环境信息。

## 功能目标

多源异构检索用于把论文、预印本、代码、模型、数据集、基准/榜单、相关网址、本地文件和内部接口中的资料统一检索出来，并转成可导入 Zotero 文库的候选条目。

核心流程是：

```text
输入主题词或自然语言需求
  -> 选择主题词检索 / 自然语言检索 / 智能体检索（实验）
  -> 主题词直连多源，或由 V4 planner 生成源端适配 query
  -> 自然语言路线中由用户确认 / 编辑计划
  -> 调用多源 provider
  -> 标准化候选条目
  -> 去重、融合、多源命中标记
  -> 规则评分或 AI 推荐排序
  -> 用户确认后导入 Zotero 副本文库
```

系统不会把检索结果直接写入文库。所有结果都会先进入候选区，由用户确认后再导入。

## 已实现能力

### 1. 多源检索

当前支持的公开源和内部源包括：

- Crossref
- arXiv
- PubMed
- Semantic Scholar
- DataCite
- Europe PMC
- DBLP
- OpenReview
- Figshare
- OSF
- OpenML
- GitHub
- GitLab
- HuggingFace
- Zenodo
- Brave Search
- Local CSV / JSONL
- HTTP JSON
- SQLite
- Object Manifest

不同数据源返回格式差异很大。后端通过统一 provider 协议把这些来源转换为统一候选结构，再交给前端展示和导入流程。

### 2. 统一候选模型

每个检索结果会被标准化为候选条目，主要字段包括：

- 标题、作者、年份、摘要
- DOI、PMID、arXiv ID、ISBN 等标识符
- 来源 source、命中来源 sources、多源命中数量 source_count
- 资料类型 item_type
- 资源类型 resource_type / resource_type_tags
- 来源类别 source_category
- 自定义源实例 source_instance_id
- 质量分层 quality_bucket
- 网页元数据 webpage_metadata
- landing URL / PDF URL
- Zotero 导入预览 item
- 检索证据 evidence
- 规则评分或 AI 评分 ai_evaluation

候选结构统一后，后续去重、排序、评分、导入都可以复用同一套逻辑。

### 3. 去重和多源融合

后端会基于强标识符和弱匹配规则合并重复候选：

- 强标识符：DOI、PMID、PMCID、arXiv ID、ISBN、ADS Bibcode
- 弱匹配：规范化标题、年份、第一作者

合并后会保留多源命中信息，并补齐来自不同来源的 URL、PDF、摘要和标识符。

### 4. AI 深度检索

用户可以输入自然语言需求，让 V4 planner 生成源端适配的多条检索词。计划结果会包含：

- 按资源类型拆分的 query group
- 面向英文源的英文 query，以及按源能力保留的中文 / 双语 query
- 检索意图
- 推荐数据源
- 每条 query 的理由
- 后续批量检索所需的 query_text

自然语言检索采用两步流程：

```text
自然语言需求
  -> 生成 AI 检索计划
  -> 用户确认 / 编辑每组 query
  -> 开始后台 guided job
```

用户可以在计划中修改每组 query，删除不需要的组，也可以补充新的检索组。确认后才会创建后台 guided job。任务执行后，用户切换页面、刷新或进入其他功能区时，后端任务仍会继续运行；回到检索页后，前端会恢复最近任务的进度或结果。

如果模型未配置，自然语言检索会明确报错并提示切换到“主题词检索”。系统不会把规则计划伪装成 AI planner 结果。

### 5. 后台任务机制

检索页中的耗时操作都已后台化：

- 快速检索：`/retrieval/search/jobs`
- AI 检索词拆解：`/retrieval/query-plan/jobs`
- 批量检索：`/retrieval/batches`
- AI 推荐排序：`/retrieval/ai-scoring-jobs`

前端只负责启动任务、轮询状态和恢复结果。后端负责实际执行。

这解决了浏览器页面切换导致任务中断的问题。需要注意的是，当前快速检索、AI 检索计划和 AI 推荐排序 job 是进程内后台任务；页面切换不会中断，但服务进程重启会丢失未完成的内存任务。已完成的检索结果仍会进入 run/candidate 记录，可恢复和导入。

### 6. AI 推荐排序

默认检索结果先使用规则评分，保证结果快速返回。用户点击“AI 推荐排序”后，系统再启动后台 AI 评分任务。

评分流程：

```text
当前候选列表
  -> 按规则置信度从高到低排序
  -> 后端逐条调用模型评分
  -> 每完成一条更新 job candidates
  -> 前端轮询并刷新候选区
  -> 只把真实 AI 推荐项加入“全选 AI 推荐”
```

AI 评分使用统一 rubric：

- topic_relevance_score：主题相关度
- metadata_quality_score：元数据质量
- source_evidence_score：来源证据强度
- import_risk_score：导入风险
- final_confidence_score：最终推荐置信度

候选卡片会明确区分：

- AI 评分
- AI 评分中
- AI 部分评分
- 规则兜底

如果模型未配置或调用失败，系统不会伪装成 AI 评分，而是显示规则兜底。

### 7. 用户确认导入

导入接口只处理用户选中的候选条目。导入时优先使用后端缓存的 `run_id + candidate_ids`，避免前端重新提交大段候选 payload。

导入后会保留：

- created / existing / conflict / failed 结果
- import evidence
- candidate provenance
- run report endpoint

这方便后续复盘“某条资料来自哪个源、为什么被推荐、什么时候导入”。

### 8. V2 引导式高覆盖检索

V2 新增“引导式检索任务”，用于把原来的“输入 query 后取每源前几条”升级为更可控的检索流程。

用户在简化界面只需要选择：

- 检索主题
- 检索模式：快速 / 高质量 / 全覆盖
- 时间范围：近 3 年 / 近 5 年 / 近 10 年 / 不限
- 资料类型：论文 / 代码 / 模型 / 数据集 / 基准/榜单 / 相关网址
- 数据源

后端会创建持久化 guided job，生成检索计划并逐条 query 执行。每完成一条 query，就把候选写入现有 `retrieval_runs` / `retrieval_candidates`，前端轮询后即可实时显示已完成部分。

guided job 持久化在应用侧 `app.sqlite`，不修改 Zotero 原生 schema。页面切换不会中断任务；服务重启后，前端读取 latest job 时可以恢复 queued / running 任务。

guided job 的 `progress` 中维护轻量事件流 `events`，用于记录任务创建、计划生成、每组检索开始、单组完成、单组失败、自动补检和最终完成状态。前端只读取现有 `guided-search-jobs` 接口返回的 job，不新增 `/search/*` 工作台接口；候选仍然写入并汇总自 `retrieval_runs` / `retrieval_candidates`。这吸收了 search-agent 分支的“任务消息流”体验，但避免引入第二套 search run / candidate pool。

三种模式默认策略：

- 快速：默认近 5 年，较少 query 和较低每源 limit，优先返回速度。
- 高质量：默认近 10 年，权威影响力排序，兼顾质量和覆盖。
- 全覆盖：默认不限时间，更多 query 和更高每源 limit；发现覆盖缺口时自动补检一次。

检索结果新增字段：

- `authority_signals`：引用数、来源库、venue、GitHub stars、HuggingFace downloads、license、多源命中等。
- `quality_score`：基于元数据完整度、强标识符、多源命中和权威信号的综合质量分。
- `coverage_tags`：paper / code / model / dataset / benchmark / website / authority / usage / full_text 等覆盖标签。
- `missing_authority_signals`：缺引用数、缺 venue、缺 stars/downloads 等可解释降级原因。

时间范围和资料类型通过统一 `SearchOptions` 传入检索层。当前实现优先做后端统一过滤，并在 source stats 的 `filtering` 字段中标明；后续可以逐源扩展为源端过滤。

覆盖报告会在页面内展示：

- 已覆盖的资料类型和数据源。
- 论文 / 代码 / 模型 / 数据集 / 基准/榜单 / 相关网址数量。
- 权威信号、多源命中数量。
- 仍可能遗漏的方向和补检建议。

### 9. V3 数据源扩展与源管理中心

V3 把“多源异构”的覆盖面从论文、代码、数据扩展为 6 类资源：

- 论文 paper
- 代码 code
- 模型 model
- 数据集 dataset
- 基准/榜单 benchmark
- 相关网址 website

新增内置源包括：

- Europe PMC：补充生物医学论文、PMID、PMCID、全文链接。
- DBLP：补充计算机科学论文、会议、期刊和作者元数据。
- OpenReview：补充 ICLR、NeurIPS 等开放评审论文和投稿记录。
- Figshare：补充数据集、软件、图表和研究对象。
- OSF：补充项目、数据、预注册和研究对象。
- OpenML：补充机器学习数据集、任务和 benchmark。
- GitLab：补充开源代码仓库，可选配置 `GITLAB_TOKEN` 提升限流。
- Brave Search：补充相关网址，需配置 `BRAVE_SEARCH_API_KEY`。

检索计划也按资源类型拆 query。引导式检索会根据用户选择的资料类型自动推荐源，例如论文优先走 Crossref / arXiv / Semantic Scholar / Europe PMC / DBLP / OpenReview，代码优先走 GitHub / GitLab / HuggingFace，数据集优先走 DataCite / Zenodo / Figshare / OSF / OpenML，相关网址走 Brave Search。

检索页新增“源管理中心”，位置仍在检索页内，不新增全局导航入口。源管理中心展示：

- 内置源是否可用、是否需要 key、最近检测状态。
- 支持的资源类型。
- 限流说明和配置入口。
- 文库级自定义源实例。

Brave 网页候选导入为 Zotero `webpage` 条目。导入前只做安全元数据抓取：title、description、OpenGraph；如果抓取超时或失败，则回退 Brave 摘要。网页抓取会阻止 localhost、私网 IP、非 http/https URL，避免把服务端变成内网探测入口。

### 10. 文库级自定义源

V3 新增文库级多实例自定义源，不修改 Zotero 原生 schema。数据保存在应用侧 `retrieval_custom_sources` 表，每个源实例包含：

- 名称
- 类型
- 启停状态
- 配置 JSON
- 最近检测结果
- 最近检测时间

第一版支持 4 类实例：

- HTTP JSON
- CSV / JSONL 文件
- Object Manifest
- SQLite

其中 SQLite 主要作为高级/旧版入口保留。旧版“每类一个配置”的 Local CSV/JSONL、HTTP JSON、SQLite、Object Manifest 仍兼容，UI 中不会强制迁移到新实例模型。

自定义源参与检索后，候选会带上 `source_instance_id`，便于区分“同一种源类型下的多个团队源实例”。

### 11. V4 三路线检索

V4 把检索页入口拆成三条路线，默认是“自然语言检索”：

- 主题词检索：直接把用户输入作为精确检索式，调用多源真实 API，不做 AI 拆词。适合用户已经知道英文主题词或标准术语的场景。
- 自然语言检索：把中文或英文描述交给 V4 planner，提取主题、约束、时间范围、资源类型、排除条件和偏好，再按源能力生成可编辑检索词组。
- 智能体检索（实验）：网页端显示说明面板但不自动启动 agent；通过仓库内 `multi-source-retrieval` skill 和 JSON CLI 给 Codex/智能体循环调用。

“快速 / 高质量”不是检索路线，而是“检索强度”。它决定时间默认值、每源候选数默认值，以及 AI planner 每个资料类型生成多少检索词：快速模式每个资料类型 3 条，高质量模式每个资料类型 5 条。AI planner 不再按数据源拆组，只按论文 / 代码 / 模型 / 数据集 / 基准 / 网址等资料类型拆组；执行时后端再把该类型的检索词分发到对应数据源。每个数据源检索多少候选由用户在页面中手动调节，当前限制为 1-50 条。代码和接口里的 `query` 指实际发给数据源 API 的检索关键词/检索式，页面文案统一称“检索词”。检索词数量和 `source_limits` 是独立概念：前者由检索强度决定，后者控制每条检索词到某个源取多少候选。

V4 planner 输出结构包括：

- `input_text`
- `detected_language`
- `normalized_topic`
- `constraints`
- `query_groups`
- `coverage_targets`
- `ambiguities`

`query_groups` 会按 `paper / code / model / dataset / benchmark / website` 和 source 拆分，每组包含：

- `resource_type`
- `source`
- `language`
- `queries`
- `must_include`
- `optional_terms`
- `exclude_terms`
- `reason`
- `confidence`

中文自然语言输入不再只做固定后缀扩展。对于 Crossref、arXiv、Semantic Scholar、GitHub、HuggingFace 等英文源，planner 必须生成英文可检索 query；对于本地、HTTP JSON、Manifest 等内部源，会按源能力保留中文或双语 query。

V4 planner 的提示词强调“资料类型级检索词”，并把源的检索风格作为背景信息传给模型：论文偏学术短语，代码偏 implementation / repository，模型偏 checkpoint / task / method alias，数据偏 dataset / benchmark。模型不需要为 Crossref、arXiv、GitHub、HuggingFace 等每个源重复生成一组 query；后端会根据资料类型自动分发到适用源。模型输出会被后端再次清洗，避免把 GitHub、HuggingFace 等平台名重复塞进检索词。

自然语言 planner 的稳定性策略是分层降级：

1. 先尝试全源计划。
2. 全源计划超时或为空时，用 compact prompt 重试。
3. 仍不可用时，按资料类型做小批重试。
4. 某个资料类型仍未被 AI 覆盖时，只给该类型生成规则补足 query。

计划中的每个 query group 都带 `planning_status` 和 `planning_message`。前端不再整体提示“AI 规划失败”，而是逐类型显示“该类型 AI 生成”或“该类型规则补足”，用户可以只检查被补足的资料类型。

### 12. 智能体工具层

V4 新增 JSON CLI 和 Codex skill，供后续“自然语言智能体检索”调用：

- CLI：`src/zotero_web_library/retrieval_cli.py`
- Skill：`skills/multi-source-retrieval/SKILL.md`

CLI 命令包括：

- `sources`：列数据源、配置状态和自定义源。
- `plan`：生成 V4 主题词或自然语言检索计划。
- `search`：调用项目 provider 执行真实多源检索并保存 run。
- `candidates`：读取 run 或 guided job 候选。
- `ai-score`：对候选执行 AI 推荐排序。
- `coverage`：生成覆盖报告。

智能体工具层只调用本项目检索 provider 和应用侧数据，不绕过项目接口，不自动导入 Zotero。

## 主要接口

| 能力 | 接口 |
| --- | --- |
| 生成 V4 检索计划 | `POST /api/library/<library_id>/retrieval/guided-search-plan` |
| 引导式检索任务 | `POST /api/library/<library_id>/retrieval/guided-search-jobs` |
| 最近引导式检索任务 | `GET /api/library/<library_id>/retrieval/guided-search-jobs/latest` |
| 引导式检索任务详情 | `GET /api/library/<library_id>/retrieval/guided-search-jobs/<job_id>` |
| 引导式候选汇总 | `GET /api/library/<library_id>/retrieval/guided-search-jobs/<job_id>/candidates` |
| 暂停 / 继续 / 停止引导式检索 | `POST /api/library/<library_id>/retrieval/guided-search-jobs/<job_id>/{pause,resume,cancel}` |
| 快速检索任务 | `POST /api/library/<library_id>/retrieval/search/jobs` |
| 最近快速检索任务 | `GET /api/library/<library_id>/retrieval/search/jobs/latest` |
| AI 检索计划任务 | `POST /api/library/<library_id>/retrieval/query-plan/jobs` |
| 最近 AI 检索计划 | `GET /api/library/<library_id>/retrieval/query-plan/jobs/latest` |
| 批量检索 | `POST /api/library/<library_id>/retrieval/batches` |
| 批量候选汇总 | `GET /api/library/<library_id>/retrieval/batches/<job_id>/candidates` |
| AI 推荐排序任务 | `POST /api/library/<library_id>/retrieval/ai-scoring-jobs` |
| 停止 AI 推荐排序 | `POST /api/library/<library_id>/retrieval/ai-scoring-jobs/<job_id>/cancel` |
| 候选导入 | `POST /api/library/<library_id>/retrieval/import` |
| 数据源状态 | `GET /api/library/<library_id>/retrieval/sources` |
| 自定义源列表/创建 | `GET/POST /api/library/<library_id>/retrieval/custom-sources` |
| 自定义源详情/更新/删除 | `GET/PATCH/DELETE /api/library/<library_id>/retrieval/custom-sources/<source_id>` |
| 自定义源预览 | `POST /api/library/<library_id>/retrieval/custom-sources/<source_id>/preview` |
| 自定义源检测 | `POST /api/library/<library_id>/retrieval/custom-sources/<source_id>/check` |
| 模型状态 | `GET /api/library/<library_id>/retrieval/model-status` |

旧的同步接口仍保留用于兼容和测试，例如 `/retrieval/search` 和 `/retrieval/candidates/evaluate`。

`POST /api/library/<library_id>/retrieval/guided-search-jobs` 的 V4 payload 可包含：

- `search_route`: `keyword` 或 `natural_language`
- `input_text`
- `planner_version`: `v4`
- `expansion_level`: 默认 `balanced`
- `language_policy`: 默认 `source_adaptive`
- `limit_per_source`: 每源检索数量，页面默认快速 5、高质量 8，可手动改为 1-50
- `source_limits`: 按数据源覆盖数量，例如 `{"crossref": 12, "github": 6, "huggingface": 4}`；执行时优先于 `limit_per_source`

## 安全边界

公开文档和代码提交不应包含：

- API Key
- GitHub / HuggingFace / Zenodo token
- 本地用户路径
- 私有数据截图
- 本地交接说明

## 设计取舍

- 检索结果不自动入库，必须由用户确认。
- 默认规则评分，AI 推荐排序由用户手动触发，避免检索变慢。
- 耗时操作后台化，页面切换不影响任务继续执行。
- V2 guided job 使用应用侧持久化表保存任务进度，不改 Zotero 原生 schema。
- 对外源失败采用单源失败隔离，避免一个 provider 出错导致整体检索失败。
- AI 失败只影响对应候选，不覆盖其他已成功的 AI 评分。
- V4 保留旧 query-plan / batch 接口作为兼容路径，但检索页主流程改用三路线 guided job。
