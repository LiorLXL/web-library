# 光明科研工作站 Light / Dark 明暗主题功能设计方案

## 1. 文档信息

- 状态：已实施并通过验收
- 日期：2026-07-22
- 目标分支：`codex/theme-light-dark`
- 关联文档：[视觉设计规范](./视觉设计规范.md)

## 2. 背景

当前系统已经具备主页、文库、多源检索、知识库、文献研读、综述写作和 API 配置等主要功能，但尚未提供统一的明暗主题切换能力。

当前主页默认使用深紫色背景，项目内工作台默认使用浅灰蓝背景和白色面板。主页与内页的视觉统一属于后续独立议题，本阶段不调整内页工作台的布局和视觉语言，只为现有页面增加可由用户主动切换、可跨页面保持的 Light / Dark 配色模式。

## 3. 本阶段决策

### 3.1 要做的内容

1. 提供 Light 和 Dark 两种主题。
2. 通过用户点击按钮进行切换。
3. 记住用户选择，刷新和跨页面跳转后继续使用同一主题。
4. 覆盖全部用户页面：
   - 主页与文库配置；
   - 文库工作台；
   - 多源检索；
   - 知识库；
   - 文献研读；
   - 综述写作；
   - API 配置。
5. 为键盘操作、屏幕阅读器和浏览器原生表单控件提供正确的主题状态。
6. 避免页面加载时出现先亮后暗或先暗后亮的明显闪烁。

### 3.2 明确不做的内容

1. 不重新设计内页工作台布局。
2. 不改变侧边导航、三栏结构、表格密度、面板尺寸和现有操作流程。
3. 不在本阶段解决主页与内页的品牌风格差异。
4. 不重构业务 JavaScript、后端接口或数据库。
5. 不改变 PDF 原始页面、用户上传图片和文献内容本身的颜色。
6. 不增加“自动”“跟随系统”作为第三个可见模式。
7. 不在服务端保存主题设置；本阶段只保存在当前浏览器。

“内页工作台主题先不动”在本文中的含义是：保持现有组件形态和信息架构，只为它们增加 Dark 状态下的语义颜色，不进行视觉改版。

## 4. 当前实现分析

### 4.1 已有基础

`static/app.css` 顶部已经定义了主色、文本色、背景色、边框、圆角、阴影、字号和间距等 CSS 变量。项目内大部分组件已经通过这些变量引用颜色，因此可以在不调整 DOM 结构的情况下增加 Dark 主题覆盖。

`reader.css` 和 `writing.css` 都依赖 `app.css` 中的公共变量，也适合沿用同一主题机制。

### 4.2 需要处理的问题

1. `app.css` 中仍存在较多十六进制、`rgb/rgba` 和 `white` 等硬编码颜色。
2. 主页颜色大部分直接写在 `.landing-*`、`.hero-*` 等规则中，没有消费公共语义令牌。
3. `reader.css` 使用了 `--color-bg-primary`，而公共令牌主要使用 `--color-bg-page`。
4. `writing.css` 使用了部分 `--color-accent-green-*` 变量，而当前公共令牌主要使用 `--color-secondary-*`。
5. 阅读器顶部栏、写作区域、弹窗遮罩、原生表单控件和部分阴影仍含白色透明背景。
6. 各模板加载的脚本并不一致。例如知识库和阅读器不依赖 `app.js`，所以不能把主题初始化只放进 `app.js`。
7. 当前模板没有公共基础模板，主题初始化和切换控件需要通过小型 Jinja 局部模板复用，避免复制逻辑。

因此，本功能不能只添加一个按钮和一组粗粒度的 Dark 覆盖；需要先补齐语义变量，再逐步替换影响主要表面的硬编码颜色。

## 5. 用户体验设计

### 5.1 默认行为

- 没有保存记录时，默认使用 Light。
- 用户点击主题按钮后，在 Light 和 Dark 之间切换。
- 用户选择写入 `localStorage`，存储键建议为 `guangming-theme`。
- 允许的存储值只有 `light` 和 `dark`；其他值一律视为无效并回退到 Light。
- 用户从主页进入任意项目页、刷新页面或重新打开浏览器后，继续使用最近一次选择。

本阶段不自动读取操作系统主题，保证界面行为可预测，并满足“通过用户点击切换”的产品要求。未来如果增加“跟随系统”，应作为第三种明确选项单独设计。

### 5.2 切换控件

- 主页：放在主页顶部品牌区右侧。
- 项目内页：放在左侧导航栏底部，避免挤占顶部业务操作区。
- 没有侧边导航的独立页面：放在页面头部右侧。
- Light 状态显示月亮图标，表示点击后进入 Dark。
- Dark 状态显示太阳图标，表示点击后进入 Light。
- 控件必须提供动态的 `aria-label` 和 `title`：
  - Light 时：“切换到深色模式”；
  - Dark 时：“切换到浅色模式”。
- 按钮必须可以通过 Tab 聚焦，并在 Enter 或 Space 时触发。

项目内切换按钮沿用现有 `.rail-btn` 尺寸、圆角和交互反馈，不引入新的导航形态。

### 5.3 动画

- 主题切换时，只对页面背景、面板、边框、文字和图标颜色使用约 `180ms` 的过渡。
- 不对所有 DOM 元素使用全局 `transition: all`，避免大型表格和阅读器重绘卡顿。
- 首次加载不播放主题过渡，防止页面闪烁。
- 当用户启用 `prefers-reduced-motion: reduce` 时取消主题过渡。

## 6. 技术设计

### 6.1 主题状态载体

主题状态统一设置在根元素上：

```html
<html lang="zh-CN" data-theme="light">
```

切换后变为：

```html
<html lang="zh-CN" data-theme="dark">
```

CSS 只根据 `html[data-theme]` 决定语义颜色。业务组件不读取主题状态，也不在组件上增加 `dark` 类。

### 6.2 加载顺序与防闪烁

主题必须在主要样式表生效前确定。建议分为两部分：

1. `templates/_theme_head.html`
   - 放置在每个页面的 `<head>` 中，并位于 CSS `<link>` 之前；
   - 使用极小的同步脚本读取 `localStorage`；
   - 立即设置 `document.documentElement.dataset.theme`；
   - 读取失败时安全回退到 Light。
2. `static/theme.js`
   - 使用 `defer` 加载；
   - 负责绑定所有 `[data-theme-toggle]` 按钮；
   - 更新按钮图标、`aria-label` 和 `title`；
   - 切换后写入 `localStorage`；
   - 派发 `guangming:themechange` 自定义事件，为未来需要重绘的图表或编辑器预留接口。

初始化伪代码如下：

```js
const THEME_KEY = "guangming-theme";
const savedTheme = localStorage.getItem(THEME_KEY);
const theme = savedTheme === "dark" ? "dark" : "light";
document.documentElement.dataset.theme = theme;
```

实际实现应使用 `try/catch`，兼容禁用本地存储或隐私模式下的异常。

### 6.3 语义令牌

品牌色与页面表面色需要分开。组件只能使用语义令牌，不应根据主题直接写一套组件颜色。

建议保留并补齐以下类别：

| 类别 | 建议变量 | 用途 |
|---|---|---|
| 页面背景 | `--color-bg-page` | 页面最底层画布 |
| 面板背景 | `--color-bg-panel` | 顶栏、侧栏、主要面板 |
| 卡片背景 | `--color-bg-card` | 卡片、输入区、弹窗 |
| 次级表面 | `--color-surface-tint` | 工具区、禁用区、浅色分组 |
| 悬停表面 | `--color-bg-hover` | 列表和按钮悬停 |
| 主文字 | `--color-text-primary` | 标题、正文、主要数值 |
| 次文字 | `--color-text-secondary` | 描述、次级标题 |
| 弱文字 | `--color-text-muted` | 提示、元数据、占位符 |
| 默认边框 | `--color-border-default` | 面板和卡片边界 |
| 弱边框 | `--color-border-subtle` | 表格分割线、次级边界 |
| 遮罩 | `--color-overlay` | 弹窗背景遮罩 |
| 阴影色 | `--color-shadow-rgb` | 统一生成主题阴影 |
| 选中文本 | `--color-selection-bg` | 文本选择背景 |

为兼容现有样式，第一阶段补充别名：

```css
--color-bg-primary: var(--color-bg-page);
--color-accent-green: var(--color-secondary);
--color-accent-green-dark: var(--color-secondary-dark);
--color-accent-green-light: var(--color-secondary-light);
--color-accent-green-soft: var(--color-secondary-soft);
--color-accent-green-rgb: var(--color-secondary-rgb);
```

别名用于降低一次性修改风险；后续可以逐步收敛为唯一命名。

### 6.5 主页处理原则

本阶段不统一主页与内页的品牌风格，但主页也必须支持两种模式：

- 当前深紫色外观作为 Dark 版本的基础；
- Light 版本继续保留紫色与粉色品牌色，但把页面画布、卡片和正文切换为浅色表面；
- 轨道、粒子、渐变标题和 Hero 布局保持不变；
- 将主页硬编码颜色收敛为局部语义变量，例如：
  - `--landing-bg`；
  - `--landing-surface`；
  - `--landing-border`；
  - `--landing-text`；
  - `--landing-muted`；
  - `--landing-accent`。

这样可以完成明暗切换，同时避免提前进入主页视觉改版范围。

### 6.6 内页工作台处理原则

- Light 下尽量保持当前视觉不变，避免本功能造成无关回归。
- Dark 下只替换表面、文字、边框、阴影和控件状态颜色。
- 保持现有导航宽度、卡片圆角、间距、按钮形状和表格密度。
- 语义标签、评分星级、附件类型、成功/危险状态不得只依赖颜色表达含义。
- 表格斑马纹、选中行、悬停行需要在 Dark 下保持三种不同层级。
- 禁用状态不能仅降低透明度到难以阅读，应同时调整背景、边框和文字颜色。

### 6.7 阅读器和写作页特殊规则

#### 文献研读

- PDF 页面画布保持原始白色，不做 CSS 反色。
- 阅读器外壳、目录、智能体面板、顶部栏和浮动工具栏跟随主题。
- 文本选择、批注高亮和下划线必须在两种主题下可见。
- PDF.js 渲染结果不参与主题转换。

#### 综述写作

- 写作面板、步骤导航、对话区域和表单控件跟随主题。
- 编辑器的光标、选区、占位符和禁用状态需要单独检查。
- 导出内容不携带界面主题颜色，保证文档生成结果不受 Dark 模式影响。

## 8. 分阶段实施计划

### 阶段一：基础设施

1. 添加根元素主题状态和早期初始化。
2. 添加主题切换脚本与公共按钮模板。
3. 在所有页面接入主题资源。

5. 添加主题持久化和无 `localStorage` 时的降级逻辑。

### 阶段二：公共令牌

1. 补齐缺失变量和兼容别名。
2. 定义 Dark 语义令牌。
3. 替换页面背景、面板、文字、边框、阴影和输入框中的关键硬编码颜色。
4. 统一浏览器原生表单控件、滚动条和文本选择颜色。

### 阶段三：逐页适配

建议按以下顺序处理，便于从公共组件向特殊页面扩展：

1. 文库工作台与 API 配置；
2. 多源检索与知识库；
3. 文献研读；
4. 综述写作；
5. 主页。

### 阶段四：测试与收尾

1. 补充静态契约测试。
2. 执行 JavaScript 语法检查和前端相关测试。
3. 在 Light / Dark 下逐页检查。
4. 修复残留白块、低对比度文字和不正确阴影。
5. 更新视觉设计规范。

## 10. 无障碍与质量要求

- 普通正文与背景的对比度目标不低于 `4.5:1`。
- 大号文字、图标和控件边界的对比度目标不低于 `3:1`。
- 焦点样式在 Light / Dark 下都必须清晰可见。
- 成功、警告、危险和选中状态不能只使用颜色表达。
- 浏览器原生输入框、下拉框和滚动条应通过 `color-scheme` 与当前主题一致。
- 不允许通过 `filter: invert()` 对整个页面实现 Dark。
- 不允许为了 Dark 模式复制整套组件 CSS。
- 不允许在业务 JavaScript 中逐个设置组件颜色。

## 11. 主要风险与规避方式

| 风险 | 表现 | 规避方式 |
|---|---|---|
| 初始化过晚 | 页面先白后黑 | CSS 前设置根元素主题 |
| 硬编码颜色遗漏 | Dark 中出现突兀白块 | 按背景、面板、文字、边框分类扫描 |
| 深色阴影不自然 | 面板层级消失或出现黑边 | Dark 使用更轻的外阴影和更明确的边框 |
| 状态色对比不足 | 标签和提示难以阅读 | 为 Dark 单独调整状态色和柔光背景 |
| PDF 被错误反色 | 论文内容失真 | 只改变阅读器外壳，不改变 PDF 画布 |
| 大型页面切换卡顿 | 表格和矩阵明显掉帧 | 限制过渡属性和作用范围 |
| 本地存储不可用 | 初始化脚本报错 | 所有读写使用 `try/catch`，回退 Light |
| 多模板接入不一致 | 部分页面无法切换 | 使用公共局部模板和契约测试 |

## 12. 验收标准

满足以下条件即可认为本功能完成：

1. 用户可以在所有主要页面通过点击切换 Light / Dark。
2. 主题选择可以跨页面、刷新和浏览器重启保持。
3. 没有保存设置时稳定回退到 Light。
4. 页面加载时没有明显的主题闪烁。
5. Light 下内页工作台与改动前基本一致。
6. Dark 下不存在大面积错误白块、不可读文字或消失的边框。
7. PDF 内容不被反色，写作导出结果不受主题影响。
8. 切换控件支持键盘和屏幕阅读器。
9. 全部现有测试和新增主题契约测试通过。
10. 本阶段没有改变内页布局、业务流程和后端接口。

## 13. 实施结果
